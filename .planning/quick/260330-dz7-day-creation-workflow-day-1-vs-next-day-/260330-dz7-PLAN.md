Extend `createDaySchema` to accept an optional `batchDay`:
```typescript
const createDaySchema = z.object({
  batchStrainId: z.string(),
  batchDay: z.number().int().min(1).optional(),
  notes: z.string().max(500).optional(),
})
```

After the existing `lastDay` / `nextDay` auto-increment logic, add override logic:
```typescript
const requestedDay = parsed.data.batchDay ?? nextDay
```

Before `prisma.day.create`, add a conflict check:
```typescript
const conflict = await prisma.day.findFirst({
  where: { batchId: id, batchStrainId, batchDay: requestedDay },
})
if (conflict) {
  return NextResponse.json(
    { error: `Day ${requestedDay} already exists for this strain in this batch.` },
    { status: 409 }
  )
}
```

Then use `requestedDay` instead of `nextDay` in `prisma.day.create`:
```typescript
data: { batchId: id, batchStrainId, batchDay: requestedDay, notes }
```

The existing auto-increment path (`lastDay`/`nextDay` query) should remain — it is still needed to compute the fallback when `batchDay` is not provided.
