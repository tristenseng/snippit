## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, inline chip spacing |
| sm | 8px | Compact form field gaps, button icon-to-label gap |
| md | 16px | Default form field vertical gap, card padding, table cell padding |
| lg | 24px | Section heading-to-content gap, card-to-card gap |
| xl | 32px | Page section separation |
| 2xl | 48px | Major content block breaks |
| 3xl | 64px | Page-level vertical padding (desktop only) |

Exceptions:
- Touch targets: minimum 44px height on all interactive elements (buttons, inputs, autocomplete items, table row actions) — enforced via `min-h-[44px]` per existing Phase 1 pattern
- Manager desktop views may use tighter table row heights of 40px when data density requires it, provided hover/focus targets expand to 44px

Source: Existing Tailwind usage in Phase 1 components + `minHeight.touch: 44px` in tailwind.config.js.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / Data / Label | 14px (sm) | 400 (normal) | 1.5 |
| Heading / Display / CTA | 20–24px (xl–2xl) | 600 (semibold) | 1.2 |

Notes:
- Input fields use `text-base` (16px) on mobile to prevent iOS zoom on focus — existing LoginForm pattern (`text-base sm:text-sm`). All `<input>` and `<select>` elements MUST follow this pattern.
- Table/list data rows use Body role (14px/400).
- Section headings (e.g. "Active Batch", "Employee Accounts") use 20px/600.
- Page-level titles (e.g. "Batch Management", "Admin Panel") use 24px/600.
- CTA button text uses 14–16px/600 (semibold is sufficient; 700/bold is not used in this phase).
- The weight scale is exactly 2: 400 (body, data, labels, secondary text) and 600 (headings, display titles, CTAs).

Source: Observed from globals.css (16px base), LoginForm.tsx label/input patterns, dashboard layout `text-xl font-semibold` for nav title. Weights consolidated from prior draft per checker feedback (max 2 weights).

## Color

| Role | Tailwind Token | Hex Equivalent | Usage |
|------|---------------|----------------|-------|
| Dominant (60%) | gray-50 | #F9FAFB | Page background (`min-h-screen bg-gray-50`) |
| Secondary (30%) | white + gray-200 borders | #FFFFFF / #E5E7EB | Cards, nav bar, sidebar, form containers, table backgrounds |
| Accent (10%) | blue-600 | #2563EB | Reserved elements listed below |
| Destructive | red-600 | #DC2626 | Destructive actions only |

Accent (blue-600) reserved for:
1. Primary CTA buttons — "Create Batch", "Add Day", "Add Entry", "Add Employee", "Submit Day"
2. Focus rings on all form inputs (`focus:ring-blue-500 focus:border-blue-500`)
3. Active/selected state in autocomplete dropdown
4. Links in breadcrumb navigation

Accent is NOT used for:
- Secondary actions (use gray-100 bg / gray-700 text)
- Table headers
- Status badges (use semantic green/yellow/gray)
- Disabled states (use blue-400 per existing LoginForm pattern)

Additional semantic colors:
- `green-100 / green-800` — ACTIVE batch status badge, submitted day indicator
- `yellow-100 / yellow-800` — INACTIVE batch status badge, unsubmitted day indicator
- `gray-100 / gray-500` — COMPLETED batch status badge, deactivated account indicator
- `red-100 / red-700` — Destructive action confirmation banners, deactivated account warning

Source: Observed from dashboard layout (`bg-gray-50`, `bg-white`, `border-gray-200`), LoginForm (`bg-blue-600`, `text-red-600`), MobileNav (`text-gray-600`, `hover:bg-gray-100`).

## Component Inventory

These are the new UI components needed for this phase. No shadcn — implement as raw Tailwind + React.

### Manager: Batch Management

**BatchStatusBadge** — inline pill showing INACTIVE / ACTIVE / COMPLETED
- ACTIVE: `bg-green-100 text-green-800 text-xs font-medium px-2 py-0.5 rounded-full`
- INACTIVE: `bg-yellow-100 text-yellow-800 ...`
- COMPLETED: `bg-gray-100 text-gray-500 ...`

**BatchCard** — card showing batch number, location, status badge, strain list, and primary action
- Container: `bg-white rounded-lg shadow-sm border border-gray-200 p-4`
- Action buttons anchored to card footer

**DayList** — ordered list of batch days with submitted/unsubmitted indicators and row-level actions
- Table on desktop (`hidden sm:table`), stacked cards on mobile (`sm:hidden`)

**EmployeeAutocomplete** — search input with live dropdown
- Input: standard text input with `text-base sm:text-sm min-h-[44px]`
- Dropdown: `absolute z-10 w-full bg-white border border-gray-200 rounded-md shadow-lg mt-1`
- Each result row: `px-3 py-2 text-sm hover:bg-blue-50 cursor-pointer min-h-[44px] flex items-center`
- No-results state: `px-3 py-2 text-sm text-gray-400 italic`

**WeightEntryRow** — single employee weight entry in the day detail form
- Inline layout: avatar/name | grams input | hours input (optional) | delete icon button
- Edit state: inputs active; display state: read-only text with edit icon

**StrainSelector** — dropdown or radio group for selecting active strain within a day session
- Uses native `<select>` on mobile; custom dropdown acceptable on desktop if touch targets met

### Admin: User Management

**UserTable** — paginated list of user accounts
- Columns: Name | Role badge | Location(s) | Status (active/deactivated) | Actions
- Mobile: stacked card layout, not table

**UserRoleBadge** — EMPLOYEE / MANAGER / ADMIN pill
- ADMIN: `bg-purple-100 text-purple-800`
- MANAGER: `bg-blue-100 text-blue-800`
- EMPLOYEE: `bg-gray-100 text-gray-700`

**CreateAccountModal** — modal dialog for admin to create new employee account
- Standard modal: `fixed inset-0 z-50 flex items-center justify-center`
- Backdrop: `bg-black bg-opacity-25`
- Panel: `bg-white rounded-lg shadow-xl p-6 w-full max-w-md mx-4`

**DeactivateConfirmDialog** — confirmation dialog for soft-deleting an account
- Uses the same modal pattern as CreateAccountModal

### Shared

**FormSection** — labeled section wrapper for multi-field forms
- `<fieldset>` with `<legend class="text-sm font-medium text-gray-700 mb-2">`

**ActionButton variants**:
- Primary: `bg-blue-600 hover:bg-blue-700 text-white font-semibold rounded-md px-4 py-2 min-h-[44px]`
- Secondary: `bg-white hover:bg-gray-50 text-gray-700 border border-gray-300 font-semibold rounded-md px-4 py-2 min-h-[44px]`
- Destructive: `bg-red-600 hover:bg-red-700 text-white font-semibold rounded-md px-4 py-2 min-h-[44px]`
- Ghost/icon: `p-2 text-gray-400 hover:text-gray-600 hover:bg-gray-100 rounded-md min-h-[44px] min-w-[44px]`

**InlineAlert** — non-modal contextual feedback
- Success: `bg-green-50 border border-green-200 text-green-800 text-sm rounded-md px-4 py-3`
- Error: `bg-red-50 border border-red-200 text-red-700 text-sm rounded-md px-4 py-3 role="alert"`
- Info: `bg-blue-50 border border-blue-200 text-blue-800 text-sm rounded-md px-4 py-3`

## Page Layout Map

| Route | Primary User | Layout | Focal Point |
|-------|-------------|--------|-------------|
| `/batches` | Manager | List of batch cards; "Create Batch" CTA in page header | "Create Batch" primary button in top-right of page header — the dominant call to action when the list is empty or growing |
| `/batches/[id]` | Manager | Batch detail: header with status + strain list; day list below | The day list itself — specifically the topmost unsubmitted day row, which is where the manager's next action lives |
| `/batches/[id]/days/[dayId]` | Manager | Day detail: strain selector + weight entry form + submitted entries list | The EmployeeAutocomplete input — it is the primary data-entry trigger for every weight recording action on this screen |
| `/admin/users` | Admin | User table with search/filter; "Add Employee" CTA in page header | "Add Employee" primary button in top-right of page header — the dominant call to action for account provisioning |
| `/admin/users/[id]/edit` | Admin | Edit form for user profile (name, role, locations) | The Role select field — changing role is the highest-consequence edit action on this form |

All routes live inside the existing `(dashboard)` layout (`max-w-7xl mx-auto py-6 px-4 sm:px-6 lg:px-8`).

Manager data entry views (`/batches/**`) are desktop-primary — forms are full-width at `max-w-2xl` on desktop, full-width on mobile.

## Interaction Contracts

### Autocomplete (EmployeeAutocomplete)
- Debounce: 200ms after keystroke before filtering
- Minimum characters: 1 (first-name prefix match)
- Filter scope: active employees only (`deactivatedAt IS NULL`)
- Keyboard: arrow keys navigate list, Enter selects, Escape dismisses
- After selection: input clears, selected employee name appears as a tag above the grams field
- If employee already added to today's entry for this strain: show inline warning, do not allow duplicate

### Weight Entry Form (day detail)
- Strain must be selected before employee search is enabled
- Grams field: numeric input, max 4 digits before decimal, 1 digit after (e.g. `999.9`), placeholder `0.0`
- Hours field: numeric input, nullable, placeholder `—` (displayed as optional label)
- "Add Entry" button disabled until both employee is selected and grams > 0
- On add: row appears in submitted entries list below the form; form resets for next employee
- Inline edit: clicking edit icon on a row puts that row's inputs back into the form (inline edit, not modal)
- Delete: ghost icon button with single-click confirm via inline `"Remove?"` + "Yes, Remove" / "Keep Entry" text links (no modal for this action)

### Batch Creation
- Form fields: Strain(s) multi-select (from existing strain list or create new inline), Location (pre-filled from manager's location, read-only)
- Batch number: auto-assigned (per-location increment, shown as read-only preview in form)
- Validation: at least one strain required, strain name max 50 chars
- Submit creates batch in INACTIVE status; separate "Activate" action on batch detail

### Day Submission
- "Submit Day" button appears on day detail view once at least one weight entry exists
- Confirmation: inline prompt "Submit Day {N}? Employees will see this data." with "Submit Day" (primary) and "Keep Editing" (ghost) buttons — no modal
- After submission: day row gains green "Submitted" badge; entries remain editable (no lock per CONTEXT.md decisions)

### Admin: Account Creation
- Modal opens via "Add Employee" button
- Fields: First name, Last name, Email, Role (select: EMPLOYEE / MANAGER / ADMIN), Location assignment (multi-select for employees, single for managers)
- Email is set on creation and not shown as editable anywhere
- On create: account created with `forcePasswordReset = true`; success InlineAlert shown in modal before close

### Admin: Account Deactivation
- "Deactivate" link in user table row actions
- Opens DeactivateConfirmDialog: "Deactivate [Name]? Their historical data will be preserved. They will no longer be able to log in."
- Two buttons: "Deactivate Account" (destructive red) and "Keep Account" (ghost)
- After deactivation: row shows "Deactivated" status badge in gray; row is not removed from table

## Accessibility Contract

- All form inputs have associated `<label>` elements (not just placeholder text)
- Error messages use `role="alert"` for screen reader announcement
- Modal dialogs trap focus and return focus to trigger element on close
- Autocomplete dropdown uses `role="listbox"` with `role="option"` items and `aria-activedescendant`
- All icon-only buttons have `aria-label`
- Status badges include text content (not color alone) to communicate state
- Minimum touch target: 44px height on all interactive elements
