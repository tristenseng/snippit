## Integration Findings

### Finding 1 — Deactivated Users Can Still Log In [HIGH]

**File:** [src/lib/auth.ts:36-46](src/lib/auth.ts#L36-L46)
**Affects:** ADMIN-01, AUTH-02

The `CredentialsProvider.authorize` callback validates email and bcrypt password but does not check `deactivatedAt`. A deactivated user with knowledge of their password can receive a valid JWT session and access all dashboard routes.

The `deactivatedAt` field is correctly set by the deactivate API, filtered out by the employee search API, and displayed in the admin UI — but there is no login gate.

**Fix required:** Add `if (user.deactivatedAt) return null` in the authorize callback.

---

### Finding 2 — Weight Entry Edit Row Disappears After Save [MEDIUM]

**File:** [src/components/days/WeightEntryForm.tsx:272-275](src/components/days/WeightEntryForm.tsx#L272-L275)
**Affects:** DATA-03

`WeightEntryRow.onUpdated` is wired to remove the entry from parent state (`setEntries(prev => prev.filter(...))`). This is correct for DELETE but wrong for PATCH — after editing an entry's grams/hours and saving, the row disappears from the UI until the page reloads. The data is correctly persisted to the database; only the local state update is wrong.

**Fix required:** Change the `onUpdated` callback to update the entry in state rather than remove it, or pass the updated entry data back from `WeightEntryRow`.

---

### Finding 3 — Middleware Matcher Does Not Cover API Routes [LOW]

**File:** [src/middleware.ts:49-56](src/middleware.ts#L49-L56)
**Affects:** AUTH-04 (defense-in-depth)

The middleware `config.matcher` covers `/admin/:path*`, `/batches/:path*`, `/dashboard/:path*`, `/set-password`, and `/api/protected/:path*` — but not `/api/admin/*`, `/api/batches/*`, `/api/strains/*`, or `/api/employees/*`. Auth for these routes is correctly handled route-level via `getServerSession`, but there is no middleware-level fallback. A future route that omits `getServerSession` will be completely unprotected.

**Action:** Not a blocker today; address in a hardening phase.

---

### Finding 4 — Admin Pages Check `role` Not `activeRole` [LOW]

**Files:** [src/app/(dashboard)/admin/users/page.tsx:14](src/app/(dashboard)/admin/users/page.tsx#L14), admin strains, admin locations, admin edit page
**Affects:** AUTH-04

All four admin server-side pages gate access with `session.user?.role !== 'ADMIN'` (permanent role), while all Phase 02 API routes use `ROLE_PERMISSIONS[activeRole ?? role]` (effective role). Under current use this is consistent — admin APIs still accept admin users regardless of role switching — but it's logically inconsistent with the role-switching design and would break if the switching semantics change.

---

### Finding 5 — Misleading Reactivation Message in EditUserForm [LOW]

**File:** [src/app/(dashboard)/admin/users/[id]/edit/EditUserForm.tsx:118](src/app/(dashboard)/admin/users/%5Bid%5D/edit/EditUserForm.tsx#L118)
**Affects:** ADMIN-01 (UX only)

The edit form shows "Reactivation is not yet supported" for deactivated users. Reactivation IS implemented (`/api/admin/users/[id]/reactivate` and the Reactivate button in UserTable). The message is misleading — it should direct the admin back to the user table.
